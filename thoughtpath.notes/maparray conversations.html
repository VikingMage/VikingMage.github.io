🧪 Example: Minimal JS Modular Core (~30–50 LOC)

const registry = {};

function register(name, plugin) {
  if (registry[name]) throw new Error("Plugin already registered");
  registry[name] = plugin;
  plugin.init?.();
}

function use(name, ...args) {
  const plugin = registry[name];
  if (!plugin) throw new Error("Plugin not found");
  return plugin.run?.(...args);
}

// Example plugin
register("logger", {
  init() { console.log("Logger ready"); },
  run(msg) { console.log("LOG:", msg); }
});

This covers:

  • A central plugin registry
  • register() with safety
  • use() with runtime execution

in js, can you both throw error and return false?

In JavaScript, you cannot both throw an error and return false in the same execution path, because as soon as the throw statement is executed, the function immediately exits, and any code after it (like return false) will not run.

🚫 This will not work as intended:

function example() {
  throw new Error("Something went wrong");
  return false; // Unreachable — this line is never executed
}

✅ You can do one or the other, depending on your logic:

Option 1: Throw an error

function mustSucceed(input) {
  if (!input) {
    throw new Error("Input is required");
  }
  return true;
}

Option 2: Return false instead of throwing

function trySomething(input) {
  if (!input) {
    return false; // Fail silently
  }
  return true;
}

✅ If you want to allow both, use a try/catch wrapper to return false from the caller:

function riskyOperation() {
  throw new Error("Boom");
}

function safeWrapper() {
  try {
    riskyOperation();
    return true;
  } catch (e) {
    console.error(e);
    return false;
  }
}

This pattern is common when you're handling failure modes gracefully without letting the error propagate.

🧩 Why splice() is performant

✅ 1. In-Place Mutation

  • splice() does not create a new array — it modifies the original array in place.
  • This avoids memory allocation and copying costs that methods like filter() or slice() introduce.
array.splice(start, deleteCount, ...itemsToInsert);

✅ 2. Tight Native Implementation

  • JavaScript engines (like V8) heavily optimize splice() at the C++ level.
  • It has fewer abstraction layers compared to manual operations (e.g., shifting manually in loops).

✅ 3. Flexible and Atomic

  • You can remove, insert, or replace elements in a single operation.
  • Compared to splitting with slice() and stitching back together, splice() is cleaner and faster.
const arr = [1, 2, 3, 4];
arr.splice(1, 2, 'x'); // => [1, 'x', 4]

✅ 4. Avoids Creating Intermediate Arrays

  • slice() and concat() both create new arrays.
  • If you're mutating large arrays frequently, that leads to GC churn and pressure.
  • splice() lets you re-use the same memory footprint, which improves cache locality and runtime efficiency.

⚠️ Caveats

  1. Shifts elements:

    • Deleting or inserting at the start of a large array causes a shift of all following elements, which can be costly in some edge cases.

  2. Performance impact is positional:

    • Deleting from the end of an array: ✅ fast
    • Deleting from the start of a large array: ❌ slower (because it must shift every index)

✅ When to Prefer splice()

Use Case Is splice() good?
Remove elements in-place ✅ Yes
Insert items at specific index ✅ Yes
Replace segment with new items ✅ Yes
Preserve same array object ✅ Yes
Remove from start of massive array ⚠️ Maybe (check shift cost)
Want immutability or non-mutating op ❌ No, use slice() instead
